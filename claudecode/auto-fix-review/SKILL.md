# 自动化修复 Review

当用户说"自动化修复review"或执行 `/auto-fix-review` 时，执行以下流程：

## 首次执行

1. 扫描项目根目录下的 `review/` 目录（如不存在则扫描 `reivew/`）
2. 找出所有 `第X次Review.md` 文件
3. 对比是否有对应的 `第X次Review修复总结.md`
4. 对未处理的 Review 文件：读取 → 修复 → 写入修复总结
5. 输出修复摘要

## 持续监控

首次执行完成后，使用 ScheduleWakeup 设置 60 秒后唤醒，持续轮询新文件：

```
ScheduleWakeup(delaySeconds=60, reason="轮询新 Review 文件", prompt="自动化修复review")
```

每次唤醒时：
1. 扫描是否有新的 Review 文件
2. 有新文件 → 处理并输出摘要
3. 无新文件 → 静默，再次 ScheduleWakeup 60 秒后唤醒

用户说"停止监控"时，不再 ScheduleWakeup，结束轮询。

## 修复规则

- 🔴 高严重度：必须修复
- ⚠️ 中严重度：根据实际情况判断，尽量修复
- ⚠️ 低严重度：一般不修复，记录到总结中
- 未修复的问题需记录原因

## 修复总结格式

```markdown
# 第X次 Review 修复总结

**修复日期**: YYYY-MM-DD
**审查文件**: `review/第X次Review.md`

## 一、已修复问题
（表格：问题、严重程度、修复方式、状态）

## 二、未修复问题
（表格：问题、严重程度、原因）

## 三、修复文件清单
（表格：文件、修复内容）

## 四、验证建议
（可选）
```

## 注意事项

- 无新文件时不做任何输出
- 修复后输出摘要即可，不需要用户确认
